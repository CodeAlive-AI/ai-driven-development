# Outside-In TDD Knowledge

Core concepts for test-first development: walking skeleton, vertical slice, outside-in flow, characterisation tests, and the safety net of a test suite.

## Overview

Outside-in TDD starts tests at the system boundary (HTTP, CLI, message queue) and works inward. The first goal is a Walking Skeleton: a thin vertical slice from data ingress to data persistence that ships working software early. Tests act as a driver of change, analogous to double-entry bookkeeping — they force the production code to justify itself.

## Key Concepts

### Vertical Slice

**Definition**: A feature implemented end-to-end — from the outer boundary (e.g. an HTTP endpoint) all the way down to data persistence — using the simplest possible code at each layer.

The point is early feedback on the entire development lifecycle and getting something deployable into production.

**Key points**:
- Pick the simplest feature you can think of, and prefer data input (so later tests have data to read).
- Aim for the happy path first; note other cases for later.
- Avoid Speculative Generality — don't build frameworks before you know you need them.

### Walking Skeleton

**Definition**: A minimal, automatically deployable slice of functionality that exercises every part of the architecture without doing anything useful yet.

A Walking Skeleton gives you a test suite, a build script, a deployment pipeline, and running software — all at once, and all minimal.

### Outside-In Test-Driven Development

**Definition**: A TDD style where the first tests exercise the high-level boundary of the System Under Test; later tests work inward to finer-grained units as needed.

Starts as acceptance-test-driven development at the boundary, then transitions to unit-level tests. Boundary tests catch combinatorial explosion at the outer shell; unit tests handle edge cases.

### Characterisation Test

**Definition**: A test written after the fact that describes the behaviour of existing software, usually to protect against regressions.

Not true TDD (no red phase for new behaviour), but the right starting point for a new codebase generated by a wizard or for legacy code. Assert only superficial, stable properties — e.g. "responds with a 2xx status code" — when behaviour is expected to change soon.

### Arrange Act Assert (AAA)

**Definition**: A three-phase structure for unit tests: arrange preconditions, act on the System Under Test, assert the outcome.

The blank line between phases is a heuristic. The act phase is usually the smallest; aim for balance if you imagine rotating the test 90°.

### Short-Term vs Long-Term Memory and Legacy Code

Legacy code gets memorised into long-term memory over months. Refactoring it invalidates that memory. Better: avoid writing legacy code in the first place by keeping code simple enough to fit in short-term memory (≈ seven chunks).

### Safety Net

**Definition**: The test suite that lets you refactor production code with confidence.

Production code has a safety net (the tests). Test code does *not* have a safety net — test code can only be checked by seeing it fail against broken production code. This asymmetry dictates how you edit each kind of code.

### Triangulation

**Definition**: Adding more specific test cases to force the production code to become more generic.

"As the tests get more specific, the code gets more generic." (Robert C. Martin). The more tests you add from different angles, the better you describe the System Under Test.

## Terminology

| Term | Definition |
|------|------------|
| Walking Skeleton | Thin end-to-end slice of an app that compiles, tests, deploys |
| Vertical Slice | Feature spanning all layers, from ingress to persistence |
| Characterisation Test | Test describing behaviour of already-existing code |
| AAA | Arrange Act Assert — the three phases of a unit test |
| SUT | System Under Test — the thing the test is exercising |
| DTO | Data Transfer Object — shape-matches wire format, no behaviour |
| Domain Model | Encapsulated production type with invariants (Reservation) |
| Fake Object | Test Double with working behaviour (e.g. in-memory DB) |
| Test Spy | Test Double that records calls for later inspection |
| Humble Object | Thin wrapper over untestable infrastructure (e.g. SQL) |
| Devil's Advocate | Deliberately passing tests with obviously wrong code to expose weak tests |

## How It Relates To

- **Encapsulation**: Outside-in TDD exposes DTOs (no encapsulation) at the boundary and pushes invariants into Domain Model types reached from inside.
- **Static analysis**: Another driver of change, alongside tests. Use both; moderate static rules when they conflict with test-code readability.
- **Red Green Refactor**: The inner loop within outside-in TDD. Devil's Advocate is complementary — use both to decide when to add a test vs when to refactor.

## Common Misconceptions

- **Myth**: The first vertical slice is pointless because it just saves a hard-coded value.
  **Reality**: It establishes running software, a deployment pipeline, and a test suite. Everything else is additive from there.

- **Myth**: A unit test should have exactly one assertion ("Assertion Roulette").
  **Reality**: Assertion Roulette is specifically (a) repeatedly interleaving assert/act sections or (b) unlabelled assertions. Multiple assertions that strengthen postconditions are fine.

- **Myth**: You must strictly TDD every class.
  **Reality**: Humble Objects (DB access, UI glue) and tool-generated code (auto-generated `Equals`, `GetHashCode`) may skip TDD.

- **Myth**: You can refactor unit tests the same way as production code.
  **Reality**: Refactoring's precondition is "solid tests" — test code doesn't have that safety net, so edit tests carefully and commit independently.
