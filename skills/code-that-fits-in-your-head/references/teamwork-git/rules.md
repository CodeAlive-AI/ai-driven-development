# Teamwork and Git Rules

Concrete rules for commit hygiene, integration cadence, code review, and pull requests.

## Core Rules

### 1. Follow the 50/72 Rule for Commit Messages

The de-facto Git commit-message format. Git's own tooling (`git log --oneline`, GUIs, hosts) assumes it.

- Summary line: **no wider than 50 characters**, in the **imperative mood**.
- Leave the **second line blank**.
- Body (optional): wrap at **72 characters**. Write as much as you need.

**Imperative mood** means "Add X", "Fix Y", "Return 404 for absent reservation" — not "Added X", "Fixed Y", "Returns 404". The imperative form is usually shorter too, which helps hit the 50-char limit.

**Example** (from the book's own repo):

```
8fa3e47 Make /reservations URL segment lowercase
fbf74ae Return IDs from database in range query
033388a Return 404 Not Found for non-guid id
0f97b34 Return 404 Not Found for absent reservation
ee3c786 Read existing reservation
62f3a56 Introduce TimeOfDay struct
```

A message with a body:

```
Introduce TimeOfDay struct

This makes the roles of the constructor parameters to MaitreD clearer.
A large part of the TimeOfDay type was autogenerated by Visual Studio.
```

### 2. Explain *Why*, Not *What*

The diff already shows what changed and the code shows how it behaves. The commit message is the best place to answer:

- Why was this change made?
- Why does the code look the way it does (e.g. "most of this was autogenerated by the IDE")?

Struggling to understand the rationale behind code is one of the highest-ranked problems in software development. Use commit messages to fix that.

The summary is an exception: it *does* describe what changed, because it is used like a chapter heading when navigating history.

### 3. Make Small Commits — Every Green Is a Commit Candidate

Commit whenever the code successfully builds and tests pass. The history should be a series of snapshots of working software. Never commit code that doesn't work.

Typical micro-commit triggers:

- Rename a symbol; commit
- Extract a method; commit
- Inline a method; commit
- Add a test and make it pass; commit
- Add a Guard Clause; commit
- Fix code formatting; commit
- Add a comment; commit
- Delete redundant code; commit
- Fix a typo; commit

Not every commit will be small in practice, but tilt that way by default.

### 4. Integrate at Least Every Four Hours

Merge your work to mainline (or a shared integration branch) at least every four hours — roughly half a day's work. This is the cadence of actual Continuous Integration; the CI server is a tool, not the practice.

- If a feature takes longer than 4 hours, put it behind a feature flag and integrate the code anyway.
- More frequent is better. At the limit, integrate every time a change compiles green.
- Working only on `master` is not a substitute — the problem is concurrency, not branch names.

### 5. Keep Code Review Latency Short

Reviews should take minutes-to-hours, not days. Long latency is how bugs escape into production and how change sets grow to unreviewable sizes.

- Anchor reviews to daily rhythm: e.g. 30 min each morning and 30 min after lunch.
- If change sets are smaller than half a day of work and the team reviews twice daily, worst-case wait is ~4 hours.
- A review that takes more than an hour to conduct is not effective. Decline and ask for smaller slices instead.

### 6. Write Commit Messages (and Prose) in Proper English

Commit messages, code comments, bug replies, exception messages — these are all prose. Apply the rules of grammar, not the rules of a compiler.

- Capitalise, punctuate, proofread.
- Misspellings break text search — you'll fail to find things that are actually there.
- Dyslexia or non-native English are fine reasons to be forgiven; "I can't be bothered" is not.

### 7. Reject Change Sets That Don't Meet the Bar

Saying "no" must be a real option or review is worthless — a rubber stamp.

- Too big? Decline it. Reviewing a change that represents days or weeks of work is impractical and reviewers end up approving out of fatigue.
- Don't fall for the sunk cost fallacy: the author's time is already spent whether you accept or reject. The cost that matters is the ongoing maintenance burden of bad code.
- Rejecting 4 hours of work hurts less than rejecting 4 weeks. Another reason to keep change sets small.

### 8. Explain the "No" — Offer Alternatives

A rejection or review comment is the start of a dialogue, not a verdict:

- Don't just point out what you dislike — offer a concrete alternative.
- Cheer when you see something good.
- Remember that PR review is asynchronous writing. Tone is lost easily. Be extra polite; emojis help signal friendly intent.

### 9. One Pull Request, One Thing

Every PR must do exactly one thing. Multiple concerns → multiple PRs. In particular:

- Avoid reformatting in a substantive PR. If formatting *is* the change, do only that.
- Make sure the code builds.
- Make sure all tests pass.
- Add tests for new behaviour.
- Write proper commit messages (50/72, imperative, why).
- "As small as possible — smaller than you think."

### 10. Require a Second Pair of Eyes on Every Change

Even when a developer has permission to self-merge, team policy should require another reviewer to approve. Whether via pair/mob programming (real-time review) or PR (async review), **every change must be approved by at least two people** — that is what makes collective code ownership real.

### 11. Read PR/Review Code at Your Own Pace — Not the Author's

Sit-together reviews where the author walks the reviewer through the code are undesirable:

- The reviewer can't judge whether the code is readable on its own merits.
- The author can fast-talk over problems.

Review the code asynchronously. The author can answer questions afterwards, but they should not be narrating during the read.

## Guidelines

Less strict but recommended:

- Pair-program for hard problems; mob-program for shared learning, coaching, and onboarding.
- Rotate pair partners and code areas to prevent knowledge silos.
- Edit local Git history before pushing — keep the published trail coherent. Don't be shy about experimenting on a side branch and abandoning it.
- Use an empty initial commit (`git commit --allow-empty -m "Initial commit"`) to make future history-rewriting easier.
- Prefer the command line to a GUI at least enough to understand what Git actually does.
- Don't nitpick formatting or variable names in reviews — those can be fixed after merge (with the exception of public API names that would be breaking changes).
- Pull down the PR branch and run it on your machine if the review warrants it.

## Exceptions

When these rules may be relaxed:

- **Truly ephemeral code** (a one-off script you'll delete this afternoon): skip the Git repo, skip the process.
- **Solo work with no collaborators**: you still benefit from small commits and good messages, but the two-reviewer rule obviously can't apply.
- **Typos/formatting fixes in merged code**: chase them later rather than blocking a review.
- **A commit that truly is self-explanatory**: the body can be omitted. But this is less often than you think — when in doubt, add more context.

## Quick Reference

| Rule | Summary |
|------|---------|
| 50/72 | 50-char imperative subject, blank line, body at 72 chars. |
| Imperative mood | "Add X", not "Added X". |
| Why > what | Diff shows what; message explains why. |
| Small commits | Every green build/test is a commit candidate. |
| Integrate ≤ 4h | Merge to mainline at least every half-day. |
| Review latency | Minutes-hours, anchored to daily rhythm, not days. |
| Reject big PRs | A review that takes >1 hour isn't effective. |
| Explain the no | Offer concrete alternatives; be polite in writing. |

| Two sets of eyes | Every change needs a second reviewer. |
